# Summit XLS — Obstacle Avoidance via Reinforcement Learning

Обучение колёсного мобильного робота **Summit XLS** (mecanum-привод, 4 omni-колеса) автономной навигации к цели с обходом препятствий в симуляторе **MuJoCo**, с использованием **Soft Actor-Critic (SAC)** и лидара.

## Что это

Робот стартует в одной точке арены и должен самостоятельно доехать до цели, **не имея заранее карты** — единственный источник информации об окружении — данные 24-лучевого лидара (3 последовательных скана) и собственная одометрия. Управление — 4 независимые скорости mecanum-колёс, что даёт роботу всенаправленное движение (вперёд/назад, страйф, поворот на месте).

Агент обучается с нуля через Reinforcement Learning (SAC) по принципу постепенного усложнения задачи (**staged curriculum**): от пустой арены до арены с множеством случайно расположенных препятствий.

## Архитектура решения

| Компонент | Значение |
|---|---|
| Симулятор | MuJoCo 3.9 |
| Робот | Summit XLS (Robotnik), 4× mecanum-колесо |
| Алгоритм RL | SAC (Soft Actor-Critic), `stable-baselines3` |
| Политика | MLP 256 → 256 → 128 |
| Observation | 80 чисел: 8 навигационных + 24×3 лидар (текущий + 2 предыдущих скана) |
| Action | 4 непрерывных значения — целевые скорости колёс (рад/с, диапазон ±15) |
| Reward | Потенциальная награда за приближение к цели + штраф за столкновение (-10, мгновенное завершение эпизода) + штраф за избыточное вращение + бонус за достижение цели (+100) |

Формула вознаграждения адаптирована из:
> Choi J., Lee G., Lee C. (2021). *Reinforcement learning-based dynamic obstacle avoidance and integration of path planning.* Intelligent Service Robotics, 14, 663–677.

## Установка

### Требования

- **Python 3.10–3.13** (рекомендуется; на 3.14 в части окружений наблюдалась нестабильность `SubprocVecEnv`)
- ОС: Windows 10/11, Linux или macOS
- Минимум 8 ГБ RAM, рекомендуется 16 ГБ
- Для приемлемой скорости обучения — современный многоядерный CPU (GPU не обязателен, SAC с MLP-политикой обучается на CPU)

### Шаги

```bash
# 1. Клонировать репозиторий
git clone https://github.com/<your-username>/<repo-name>.git
cd <repo-name>/robots/summit_xl_description

# 2. Создать виртуальное окружение
python -m venv venv

# Windows:
venv\Scripts\activate
# Linux / macOS:
source venv/bin/activate

# 3. Установить зависимости
pip install -r requirements.txt
```

### `requirements.txt`

```
mujoco>=3.1.0
gymnasium>=0.29.0
stable-baselines3>=2.3.0
torch>=2.0.0
numpy>=1.24.0
```

> Все версии указаны как минимальные ориентиры — проект тестировался на `mujoco==3.9.0`. Если планируете обучать с GPU-ускорением, поставьте `torch` под вашу CUDA-версию по официальной инструкции [pytorch.org](https://pytorch.org/get-started/locally/) **до** установки остальных зависимостей.

## Быстрый старт

### Проверить, что окружение собирается и физика работает

```bash
python env_summit.py
```

Скрипт создаст среду, выполнит сброс, проедет 150 шагов вперёд и выведет в консоль показания лидара по 24 направлениям + пройденное расстояние. Если робот не сдвинулся с места — см. раздел [Диагностика](#диагностика-и-отладка).

### Запустить готовую обученную модель

В репозитории (если не исключена `.gitignore`) лежит чекпоинт `models/best_model.zip`, обученный по staged curriculum. Чтобы посмотреть на работу агента визуально:

```bash
python test_scenes.py
```

Скрипт спросит, какую модель загрузить и на какой сцене протестировать, затем откроет окно MuJoCo viewer.

## Обучение модели

### С нуля

```bash
python train.py
```

По умолчанию обучение идёт `1 500 000` шагов с автоматическим переключением сложности (см. ниже). Каждые 100k шагов сохраняется чекпоинт в `models/summit_staged_<N>_steps.zip`, лучшая по `eval`-награде модель — отдельно в `models/best_model.zip`.

### Дообучение с чекпоинта

`train.py` автоматически ищет последний `summit_staged_*_steps.zip` в `models/` и продолжает обучение с него, восстанавливая соответствующий этап curriculum. Чтобы указать чекпоинт явно — откройте `train.py` и впишите:

```python
RESUME_PATH = "models/summit_staged_600000_steps.zip"
RESUME_FROM_STEP = 600_000
```

### Параллельные среды

Число параллельных копий среды задаётся константой `N_ENVS` в начале `train.py` (по умолчанию `4`). На Linux/macOS используется `SubprocVecEnv` (настоящий параллелизм по процессам/ядрам), на Windows — автоматический безопасный откат на `DummyVecEnv` (последовательный сбор без риска зависаний при работе с MuJoCo).

### TensorBoard

```bash
tensorboard --logdir logs
```

## Curriculum Learning

Обучение разбито на 4 этапа возрастающей сложности, переключаемых автоматически по числу пройденных шагов:

| Шаги | Этап | Сцена | Что отрабатывается |
|---|---|---|---|
| 0 – 200k | 1 | Открытое поле, только внешние стены | Базовая навигация к цели |
| 200k – 400k | 2 | Узкий коридор | Избегание стен, использование лидара |
| 400k – 700k | 3 | Коридор + препятствия (50% → 100%) | Объезд препятствий в стеснённом пространстве |
| 700k – 1.5M | 4 | Открытое поле + много препятствий (50% → 100%) | Генерализация на произвольную расстановку препятствий |

Параметр `difficulty` (0.0–1.0) управляет вероятностью появления каждого препятствия на сцене и амплитудой случайного смещения его позиции — это даёт разнообразие внутри одного этапа, а не фиксированную раскладку.

## Тестирование на разных сценах

```bash
python test_scenes.py
```

Интерактивно выбирается:
1. **Модель** — лучшая сохранённая, финальная staged-модель или старая модель без лидара (скрипт сам определяет нужный класс `Env` по размеру observation space модели — 24 или 80).
2. **Сцена** — обучающая или одна из  тестовых карт.

## Диагностика и отладка

| Скрипт | Назначение |
|---|---|
| `env_summit.py` (как `__main__`) | Smoke-test: создание среды, проверка лидара, проезд по прямой |
| `debug_lidar.py` | Детальная печать каждого из 24 лучей лидара: расстояние, какой geom задет, фильтрация корпуса робота/пола |
| `debug_env.py` | Проверка базовой динамики: едет ли робот, стрейф, поворот на месте |
| `check_obstacles.py` | Проверка, что рандомизация препятствий между эпизодами действительно работает (сверка `model.geom_pos` и `data.geom_xpos`) |
| `what_is_geom11.py` | Утилита для разбора принадлежности конкретного geom к телу/группе — использовалась при отладке столкновений лидара с собственным корпусом робота |

## Известные ограничения

- На сценах, топологически сильно отличающихся от обучающих (узкие тупики, длинные лабиринты), агент может застревать — генерализация ограничена набором сцен curriculum. См. `scene_maze.xml` / `scene_zigzag.xml` как примеры сложных вне-дистрибутивных тестов.
- `summit_env_old.py` оставлен в репозитории только для воспроизведения и тестирования ранних чекпоинтов (observation space = 24, без лидара) — для новых экспериментов не использовать.
- Файлы `_tmp_stage*.xml` создаются автоматически во время обучения (одна временная сцена на процесс) и не должны попадать в репозиторий — добавлены в `.gitignore`.
